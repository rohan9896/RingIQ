- (constraint) The implementation-ready LLD/ERD inherits the approved V1 boundaries: Tenant Web App, Core SaaS Backend, Background Worker, isolated Voice AI Worker, Postgres with pgvector, object storage, and managed providers; CRUD domains remain modules rather than independent microservices.
- (decision) ADOPTED: Use one PostgreSQL database and one shared schema. Every tenant-owned record must carry tenant_id; application queries scope explicitly; PostgreSQL RLS provides defense in depth using transaction-local tenant context; tenant-aware foreign keys, indexes, retrieval filters, jobs, and object keys prevent cross-tenant association. Preserve a future extraction path for dedicated tenant storage without implementing it in V1.
- (decision) ADOPTED: Leads use canonical relational columns for stable identity and contact data plus schema-validated JSONB attributes for vertical and tenant-specific fields. Versioned category and tenant field definitions govern keys, types, validation, AI askability, filtering, and display; promote only broadly stable fields to columns later.
- (direction) Campaigns should not remain pinned to stale tenant knowledge: publishing a KB update must affect the running campaign promptly, while historical KB versions remain available for audit. Resolve whether the switch applies at call start or during an already-active call.
- (decision) ADOPTED: Knowledge content is immutable and versioned, but campaigns do not pin one KB version. Successful publication atomically changes the tenant's active KB version; each new call resolves and pins that active version for its own lifetime, and the call record stores knowledge_base_version_id. Already-active calls never switch knowledge mid-conversation. Business-profile and agent-configuration versions remain campaign-pinned.
- (decision) ADOPTED: The Core Backend is the sole writer and policy owner for product state. Voice AI Workers receive minimal signed call context and emit asynchronous idempotent events through an internal ingestion contract; persistence never blocks the media loop, transcript events may batch, finalization is retry-safe, and the transport can later move to a broker without changing event semantics.
- (decision) ADOPTED: Model reusable Lead -> Campaign Enrollment -> Call Attempt -> connected Conversation Artifacts. Enrollment owns campaign-specific eligibility, retry, progress, and outcome; each dial is a separate attempt; connected artifacts belong to the answered attempt; each attempt stores the exact call-context snapshot; current lead dashboard state is derived without overwriting historical campaign outcomes.
- (decision) ADOPTED: Use versioned category Q&A templates and structured tenant answers behind a simple Q&A plus additional-information UI. Published KB versions contain typed source records, chunks, embeddings, and source lineage; V1 exposes Q&A and manual text, while the source envelope can later add files, URLs, and integrations. Retrieval is restricted to the tenant's active published version.
- (decision) ADOPTED: Store queryable transcript segments, summaries, classifications, callbacks, knowledge gaps, statuses, and artifact metadata in PostgreSQL; store recordings, uploads, exports, and optional raw provider payloads in object storage. Tenant-prefixed object keys are private, access is Core-API-authorized through short-lived URLs, and finalized evidence is append-only/versioned rather than silently overwritten.
- (decision) ADOPTED: Keep callbacks simple in V1. Store the lead's original callback phrase, an optional normalized callback_at timestamp, timezone, source call/transcript reference, and basic workflow status; display the original phrase alongside the normalized value. Precision windows and interpretation-confidence fields are deferred until automated scheduling needs them.
- (decision) ADOPTED: Clerk owns authentication and organization identity; RingIQ stores internal users, tenants, and tenant_memberships mapped by unique Clerk IDs. Every request resolves an active internal membership for the selected organization; V1 grants all active memberships equal access, keeps RBAC dormant, and processes Clerk webhooks idempotently.
- (decision) ADOPTED: V1 lead deduplication uses tenant_id plus normalized E.164 phone number. Import rows remain immutable audit records linked to the resolved lead; duplicates reuse the lead, fill only missing data unless explicitly edited, and each campaign allows one enrollment per lead. Cross-tenant matches are unrelated.
- (decision) ADOPTED: Use PostgreSQL-backed durable jobs and transactional outbox in V1. Product mutations and required jobs commit atomically; workers claim due jobs with leases and SKIP LOCKED, enforce idempotency, bounded backoff, dead-letter states, and one active call attempt per enrollment. A broker may replace transport later without changing state transitions.
